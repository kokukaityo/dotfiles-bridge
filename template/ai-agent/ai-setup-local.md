# AI設定生成器（ローカル／リポジトリ用）

このファイルをAIエージェント（Claude Code / Codex / Gemini CLI など）に読み込ませると、
そのリポジトリ用の `AGENTS.md`（マスター）と `CLAUDE.md`（import薄ラッパー）を生成できる。

> 使い方：リポジトリ直下にこのファイルを置き、エージェントに
> 「`ai-setup-local.md` を読んで、このリポジトリの AGENTS.md と CLAUDE.md を生成して」と指示する。
> 既存ファイルがあれば差分更新でよい。

---

## 0. 前提となる設計思想（生成前にエージェントが理解すべきこと）

このリポジトリは複数AI（Claude / Codex=ChatGPT / 必要に応じGitHub Copilot・Gemini）を併用する。
設計の核は **「指示書とコンテキストをツール非依存にする」** こと。
どのツールをどの役割に挿しても運用が壊れないようにする。固定の役割分担（例：Claude=設計、Codex=実装）は
モデルの優劣が数ヶ月で入れ替わるため、覚え込ませない。

### 役割はツール名ではなく「レイヤー」で考える

1. 設計・レビュー層：要件整理、実装計画、コードレビュー。成果物は必ず `docs/plan/` に書く
2. 実装層：`docs/plan/` を読んで実装。どのエージェントでも着手できる状態を保つ
3. 雑務層：リネーム・定型修正・テスト追加。安い／クォータが余っているツールに回す

### 受け渡しは「会話履歴」ではなく「リポジトリ内のmd」で行う

- これにより、ツールを切り替えても・セッションを跨いでも・クォータが切れても作業が継続できる
- 次のエージェントが `docs/plan/` を読めば続行できる状態を常に保つ

---

## 1. 生成するファイルと構成

```
<repo>/
├── AGENTS.md      ← マスター（実体）。すべてのルールはここに書く
├── CLAUDE.md      ← 実体は @AGENTS.md の1行のみ（symlink不要・OS非依存）
└── docs/
    ├── plan/      ← 設計書・実装計画・エージェント間の受け渡し
    └── notes/     ← 作業ログ・備忘録
```

- マスターは **AGENTS.md** 側に置く（OpenAI/Google/Copilotが対応するオープン規格のため将来性が高い）
- `CLAUDE.md` は中身を `@AGENTS.md` の1行だけにする。Windowsでsymlinkが面倒なので import 方式を採用
- Gemini を使う場合は `~/.gemini/settings.json` の `contextFileName` に `AGENTS.md` を指定すれば別ファイル不要

---

## 2. AGENTS.md に必ず含める内容

エージェントは以下のセクションを生成する。`<...>` はユーザーに質問して埋める。

### 2.1 プロジェクト概要

```markdown
# Project: <プロジェクト名>

## Stack

<言語・フレームワーク・DB・主要ツール（例：Java / Spring Boot / Spring Batch / MySQL）>
```

### 2.2 エージェント間の受け渡し規約（必須・変更しない）

```markdown
## エージェント間の受け渡し規約

- 設計・実装計画は `docs/plan/<task>.md` に書く。
- 実装着手前に、必ず該当する `docs/plan/` を読むこと。
- 実装の成果物（変更概要・残課題）は同じ plan ファイル末尾に追記する。
- 会話履歴に依存しない。次のエージェントが plan を読めば続行できる状態を保つ。
- 工程（段）が変わってエージェントを引き継ぐ際は、前の窓は `/clear` してコンテキストを持ち越さない。
```

### 2.3 委譲時の出力契約（必須・変更しない）

他AIへ委譲する際、結果が呼び出し元のコンテキストを圧迫しないための契約。

```markdown
## 委譲時の出力契約

- 変更はファイルに直接書く。チャットへの全文出力は禁止。
- 返答は「変更ファイル一覧 + 要約3行以内」のみ。
- レビューは委譲先でセルフチェックさせ、重要箇所のみ呼び出し元に提示する。
```

### 2.4 タスクの形に応じた進め方（必須・変更しない）

```markdown
## 進め方の判断

- 短く密なタスクで一部だけ他AIが得意 → その窓内でオーケストレーション（委譲）。
  引き継ぎファイルを書くオーバーヘッドの方が高くつくため。
- 長い多段パイプライン（設計→実装→レビュー→テスト）→ 段の境界で `docs/plan/` 経由の引き継ぎ。
  各段でコンテキストをリセットでき、オーケストレーター役の摩耗を分散できる。
- Claude内で完結する重い探索 → サブエージェント（Task）で隔離コンテキストに逃がし、要約のみ本体に返す。
```

---

## 3. CLAUDE.md の中身（固定）

```markdown
@AGENTS.md
```

これだけ。実体を二重に持たない。

---

## 4. ツール接続の補足（生成物には含めず、セットアップ手順として案内）

- Codex を Claude Code の配下に置く場合：`.mcp.json` に
  `{"mcpServers":{"codex":{"type":"stdio","command":"codex","args":["mcp-server"]}}}` を登録し、
  `.claude/settings.local.json` の permissions で `mcp__codex__codex` 等を許可。
- 共有リポジトリでは `.mcp.json` のスコープに注意（個人検証は `--scope local` 推奨）。
- MCP設定・permissions はツールごとに概念が違うため AGENTS.md では共有しない（各ツールのconfigで個別管理）。

---

## 5. 生成手順（エージェントへの指示）

1. ユーザーに Stack（2.1）を質問する。不明な場合はリポジトリを調べて推定し、確認を取る。
2. `AGENTS.md` を生成する。2.2〜2.4 は原文のまま含める（プロジェクト非依存の普遍ルールのため）。
3. `CLAUDE.md` を生成する（中身は `@AGENTS.md` のみ）。
4. `docs/plan/` `docs/notes/` が無ければ作成する。
5. 既存ファイルがある場合は上書きせず差分提案する。
