# dotfiles-bridge の成り立ち

このドキュメントはプロジェクトの初期構想から現在に至る変遷と、その過程で下した設計判断を記録したものです。

## 始まり — AIエージェント設定の同期ニーズ

dotfiles-bridge の最初の動機は「Claude Code と Codex の AGENTS.md を、複数の PC で同じ状態に保ちたい」というシンプルなものでした。

AIエージェントの設定（AGENTS.md、スキル、カスタムコマンド）は使い込むほど洗練されていきます。しかしそれは特定のマシンのローカルに閉じた状態で育っていくため、別のマシンでは一からやり直しになります。この「育てた設定を持ち運べない」問題を解決するために、正本を1箇所に集約し symlink で各ツールに配置する仕組みを考えました。

初期構想は「AI設定の正本リポジトリ」で、対象は AIエージェント設定のみでした。

## 視野の拡大 — dotfiles 全般への汎用化

実装を進める中で、同じ仕組みがエディタ設定（VS Code の settings.json）やシェル設定（.bashrc、.zshrc）にも使えることに気づきました。

「正本を1箇所に置いて symlink で配置する」というアーキテクチャは、AI設定に限らずあらゆる設定ファイルに適用できます。対象を AI に限定する理由がなかったため、dotfiles 全般を管理する汎用ツールに方針を転換しました。コンセプトは「AI設定の正本リポジトリ」から「設定の正本リポジトリ」に拡張されました。

## 設計判断: Bash → Go への書き直し

初期実装は Bash スクリプト（`.infra/sync.sh`, `link.sh`, `setup.sh`）で始めました。プロトタイプとしては十分でしたが、以下の問題がありました。

- Windows のネイティブサポートが困難（Git Bash 依存ではポータビリティに限界）
- テストの書きにくさ（Bash のユニットテストは脆い）
- エラーハンドリングの複雑さ

Go のシングルバイナリであれば、Windows / macOS / Linux をネイティブに対応でき、テストも言語標準の仕組みで書けます。配布もバイナリ1つで済むため、dotfiles 管理ツールに適した選択でした。

設定形式もこの時点で YAML → TOML に移行しました。TOML は Go の標準的な設定形式であり、YAML のインデント依存の問題がありません。

## 設計判断: "bridge" という名前

リポジトリ名の検討は3ラウンドにわたりました。

**第1ラウンド**: `dotfiles` 単体では GitHub 上の個人 dotfiles リポジトリと区別がつかないため、サフィックスが必要と判断。

**第2ラウンド**: 機能ベースの候補（`dotfiles-sync`, `dotfiles-hub`, `dotfiles-deploy` 等）を検討。いずれも機能の一面を切り取った名前で、ツールの全体像を伝えきれませんでした。

**第3ラウンド**: 「dotfiles とはそもそも何か」に立ち返りました。dotfiles は本来、各アプリがローカルに閉じて持っている隠しファイルです。それを引っ張り出して、集めて、管理して、配る — この「本来閉じているものを橋渡しする」行為を名前にしたのが `dotfiles-bridge` です。

`sync` や `hub` が「機能の説明」であるのに対し、`bridge` は「ツールが存在する意義」を語っています。単一マシン内のアプリ間共有（アプリ間の橋渡し）も、複数マシン間の同期（マシン間の橋渡し）も、どちらも `bridge` で意味が通ります。

リポジトリ名は `dotfiles-bridge`、コマンド名は短くて打ちやすい `dotfiles` としました。この不一致は意図的で、リポジトリ名は GitHub 上で個人の dotfiles リポジトリと区別するためにサフィックスを付けています。

## 設計判断: コマンド名 `dotfiles` とディレクトリ `~/dotfiles`

コマンド名 `dotfiles` とデフォルトのデータディレクトリ `~/dotfiles` が同名のため、シェル補完時に候補が複数出る可能性があります。検討の結果、以下の理由で変更しないことにしました。

- bash と fish ではコマンドが優先され実質問題なし
- zsh でも候補が2つ出るだけで致命的ではない
- `~/dotfiles` は慣習的に定番で、より深いパスにする必要がない
- 気になるユーザーはエイリアス（`alias dfb=dotfiles`）で回避できる

コマンド名を設定で変える機能は不要と判断しました。コマンド名はバイナリのファイル名そのもので、OS/シェルの標準機能（エイリアス、シンボリックリンク）で十分対応できます。

## 設計判断: symlink 方式

設定ファイルの配置方式として、コピー方式と symlink 方式を比較しました。

**symlink 方式を選択した理由:**

- **即時反映**: アプリ側から見ると「実体ファイルがそこにある」状態なので、編集が即座に反映されます。コピー方式では編集のたびに再コピーが必要です
- **双方向性**: symlink 先を編集しても正本に届くため、どちらから編集しても同じです
- **1ファイル→複数先**: 同じ設定ファイルを複数のアプリに配置できます（例: AGENTS.md を Claude Code と Codex の両方に配置）

**トレードオフ:**

- Windows では symlink の作成に開発者モードまたは管理者権限が必要です
- 一部のアプリは symlink を正しく追跡しない可能性があります（現時点で問題は未報告）

## 設計判断: 自動 merge しない

pull 時にローカルとリモートが分岐した場合、自動 merge は行いません。ローカルの変更を `conflict/<hostname>/<timestamp>` ブランチへ退避し、既定ブランチをリモートに合わせます。

dotfiles は個人の正本です。自動 merge で意図しない内容が紛れ込むと、正本が汚れます。個人の設定ファイルなので同一行の同時編集はまず起きず、分岐が発生しても解決は数分で済みます。「安全側に倒して、必ず人が見て解決する」方針です。

## 設計判断: local モードのデフォルト化

sync.toml の `mode` はデフォルト `"local"` です。`"remote"` を明示した場合のみ origin との同期を行います。

これは「dotfiles-bridge はリモート同期ツールではない」という位置づけの表明でもあります。1台の PC 内で異なるアプリ間の設定を橋渡しするだけでも価値があり、リモートリポジトリは必須ではありません。リモート同期はオプトインの追加機能です。

これにより、Git に慣れていないユーザーでも「ローカルで設定を集約・配置する」だけの用途で安心して使えます。

## 現在と今後

2026年6月25日時点で v0.1.0-alpha.1 をタグ付けし、友人へのフィードバック収集段階にあります。主に Windows で検証しており、macOS / Linux での動作は未検証です。

alpha のフィードバックを元に、安全性・UX の改善（P1）、CI・E2E テスト・配布強化（P2）、機能拡張（P3）を段階的に進める予定です。
