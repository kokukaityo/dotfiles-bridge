## 0. 背景と北極星（最初に読む）

- 開発者は **solo**。他にも作りたいものがある。マネタイズ規模の期待は **コーヒー代〜中規模**で、まず「使われること」が最優先。
- プロジェクトの本質は dotfiles 同期そのものではなく、**複数 AI エージェント横断の個人メモリ／設定の一本化**。dotfiles 機構はその配送レイヤー。
- マネタイズ方針（確定）:
  - ツール本体は **OSS（CLI/エンジンは MIT/Apache 想定）**、広く配る。
  - 収益は **コンテンツ＋投げ銭**（Qiita を無料の入口、Zenn バッジ / GitHub Sponsors でコーヒー代、深掘りは Zenn の「本」）。
  - 売るのは「使い方（説明書）」ではなく **設計思想・運用方法論**。公開リポジトリは投げ銭を殺さない、むしろ信頼の源。
  - **`knowledge-mcp` だけは別格**：動き続けて価値を出すサーバなので、唯一「本物の有料プロダクト（オープンコア）」になり得る。ただし着手は需要シグナルが出てから。
- **判断基準**：配布インフラ・高度化は「実需が出てから」足す。先回りしない。

> 重要な状態変化: **「使ってみる」という二人目の実ユーザーが現れた**。これにより「エンジン/データ分離」のトリガー条件が満たされた。以下の分離作業に進む段階に入っている。

## 1. 確定した設計決定

### 1-1. エンジンとデータを分離する
- 現状は `.infra/`（スクリプト＝エンジン）と開発者個人の設定データが **1つの public repo に同居**している。これが諸問題の根。
- 目標:
  - **エンジン**（`.infra/` の link / sync 等のスクリプト）= **public・OSS・MIT**。fork・PR 歓迎。**個人データを含まない**。
  - **各ユーザーの dotfiles 実体** = **各自の private repo**（個人の設定データのみ）。

### 1-2. fork ではなく「テンプレート」を使う（セキュリティ上の必須事項）
- GitHub の fork はリポジトリと fork を1つの **fork ネットワーク**として commit を共有保管する。
- **CFOR (Cross Fork Object Reference)**: public repo を fork → commit → その fork を private 化・削除しても、**commit ハッシュ経由でデータに永久アクセス可能**。private repo を後から公開した場合も、公開前の commit が公開版から見え得る。GitHub が把握した上での意図的な設計。
- 個人情報が乗る dotfiles では、**fork 関係をそもそも作らないこと**が必須。
- 正しい primitive:
  - **① テンプレートリポジトリ（本命）**: 「Use this template」は fork ネットワークに属さない**独立リポジトリ**を作る。履歴非共有・private 指定可。CFOR の射程外。
  - **② 複製**: `git clone --bare` → 新 private repo へ `git push --mirror`。fork 関係なしの手動版。
- **README の導線は「Fork して」ではなく「Use this template で private リポジトリを作って」に統一する。**

### 1-3. 自動/手動の境界（事故防止）
- **データの同期 = 自動**（現状のシェル起動時 auto-sync を維持）。
- **エンジンの更新 = 手動・明示・バージョン固定**（黙って上げない）。
  - 理由: 共有物の自動更新は危険。エンジン更新で設定スキーマが変わると、黙って更新された瞬間にユーザーの設定が壊れる。

### 1-4. バージョニング（分離の安全装置）
- エンジンは必ず **tag / release** でバージョンを切る。
- データ repo 側に対応バージョンを宣言させる（`.infra-version` ファイル or `sync.yaml` の 1 フィールド）。
- これにより「新エンジン × 古いデータ repo スキーマ」の不整合を検知・防止できる。

## 2. 目標構成：エンジンの配布方法

更新反映の手段は3つ。**fork をやめても git remote / submodule / subtree はクライアント操作なので fork ネットワークに参加せず、CFOR 安全**。「GitHub の fork 関係を持たない」と「git で upstream から取り込む」は両立する。

### ① エンジンを repo に入れない（本命）
- `.infra` をユーザーのデータ repo に同居させない。エンジンは**インストールするツール**として配る:
  - インストールスクリプト（`curl ... | bash`）/ Homebrew tap / GitHub Release の tarball。
  - ユーザーの private repo には**設定データだけ**（configs / link.yaml / sync.yaml）。
  - エンジンはマシン側（例 `~/.local/bin`, `~/.local/share`）に入る。
- 更新 = `dotfile self-update` や `brew upgrade` を叩くだけ。**データ repo の git 履歴と無関係**。
- chezmoi / yadm と同じモデル（ツールは別インストール、repo は dotfiles のみ）。
- 100% shell の現状なら `self-update` サブコマンドで public エンジン repo の最新 release を取得する作りが自然。

### ② submodule / subtree（repo 内にエンジンを置きたい場合）
- **submodule**: `.infra` を public エンジン repo を指す submodule に。更新 `git submodule update --remote`。clone 時 `--recurse-submodules` が必要で初心者がつまずきやすい。
- **subtree**: `.infra/` に取り込み `git subtree pull` で更新。clone は普通だがコマンドが難解、ユーザーがエンジンを編集すると conflict。

### ③ remote 追加 + merge（最終手段）
- `git remote add engine ...` → `git pull engine main`。CFOR 安全だがエンジン履歴がデータ repo に混ざって汚れる。①②が使えるなら避ける。

> **推奨は ①。** `.infra/setup.sh` を「repo の中から実行」から「エンジンをマシンに入れる installer」へ作り替えるのが、分離への最も素直な移行。

## 3. インストーラの開発コストについて（不安への回答）
- v1 インストーラは小さい。やること:
  1. tarball か clone で取得
  2. スクリプトを `~/.local/share` に配置
  3. 起動用 1 本を `~/.local/bin` に link
  4. `self-update` サブコマンド
- 100% shell なら数十行で済む。
- 重いのは cross-platform / Homebrew / Windows / 署名 / バージョン間マイグレーション。**これらは全部あと回し可能なオプション**。該当ユーザーが現れるまで手を出さない。大変さは需要に比例して後から足す。

## 4. 動作確認の方針（UT は全面的には組まない）

このツールの本質は「symlink が張れる・git で同期される・ファイルが正しい場所に出来る」という**副作用と環境**の検証であって、純粋関数のロジック検証ではない。よって関数単位 UT より **E2E の動作確認**が効く。

### ① 使い捨て環境で本物を流す（最優先・これだけでも可）
- 分離で一番怖い失敗は「自分のマシンでは動くが、他人のクリーン環境で壊れる」。
- **Docker（`ubuntu:24.04`）** でクリーンなコンテナを立て、インストーラ → リンク → 同期 を実際に実行し結果を確認。マシンの設定汚染を受けない「他人の初回」を再現できる。数行の Dockerfile で済む。
- Mac 固有挙動は Docker（Linux）で再現不可。手元 Mac か CI の macOS runner で別途確認。

### ② bats で E2E を薄く自動化（次点）
- shell 用テストフレームワーク **bats (Bash Automated Testing System)**。
- 関数 UT ではなく「コマンドを叩いて symlink が出来てるか・中身が合ってるか」を assert する**振る舞いテスト**。
- 例: `setup` で一時 dir → インストーラ実行 → `[ -L ~/.config/foo ]` → `teardown` で掃除。
- 最初から網羅しない。**壊れたら一番痛い経路（リンク作成・sync・conflict 退避）だけ** 3〜5 本。

### ③ GitHub Actions で ①② を自動化（最後）
- push ごとに Docker(Linux) + macOS runner で bats を回す。
- **分離した瞬間に効く**: エンジン更新時に「データ repo 側のセットアップが壊れてないか」を毎回自動確認できる。手動だと必ずサボる部分なので仕組み化の価値が高い。

### 分離特有の確認
- エンジン更新時の**バージョン不整合**（新エンジン × 古いデータ repo スキーマ）を、Docker で「古い設定 + 新エンジン」の組み合わせで一度流して潰しておく。

### 最小ラインのまとめ
| タイミング | やること |
|---|---|
| 今すぐ | ① Docker でクリーン環境 E2E 手動確認 |
| 分離を出す前 | ② 壊れたら痛い 3〜5 本だけ bats |
| ユーザーが増えたら | ③ CI 化（Linux + macOS） |

> **原則: 「クリーンな環境で本当に動くか」を一度通すこと > 関数を細かくテストすること。** 順序を逆にすると、通る UT を眺めながら他人の環境で壊れる典型的失敗をする。

## 5. 次の一手（このセッションで合意した直近アクション）

1. **エンジン抽出**: `.infra/` を独立した public エンジン repo として切り出す準備（個人データを含まない形に）。
2. **installer 化**: `.infra/setup.sh` を「repo 内実行」から「エンジンをマシンに入れる installer + `self-update`」へ作り替え（方式①）。
3. **テンプレート化**: データ側を「Use this template」で配れる形に整え、README の導線を fork → template に変更。
4. **バージョニング導入**: エンジンに tag/release、データ repo に `.infra-version`（or sync.yaml フィールド）。
5. **Docker クリーン E2E**: `ubuntu:24.04` で end-to-end を一度通す Dockerfile + 手順を用意（← まずここから着手予定だった）。
6. （その後）bats で痛い経路 3〜5 本 → GitHub Actions で CI 化。

## 6. 未決・要確認事項
- 公開する「見せる用 repo」は、個人情報を抜いた **sanitized example** にする（CFOR 回避しつつ funnel として機能させる）。どこまでをサンプル化するか要設計。
- `knowledge-mcp` 側: Lambda は **方式 B（データは保存しないが通過はする）** でイメージ中。in-flight のログ・転送の扱いと compute 負担を引き受ける前提。OSS テンプレ配布（BYO クラウド）を先に出し、「立てるのが面倒」の声が出てからオーケストレーション層を検討。
- ライセンス: v1 で整える予定。エンジン本体は MIT/Apache 想定。将来 knowledge-mcp を有料化するなら、その部分だけ別 repo・別ライセンスに分離できる構造にしておく。
