## 作業ルール

- 変更対象の周辺コードと関連ドキュメントを読んでから編集する。
- ユーザー固有データやシークレットをエンジンおよび `template/` に含めない。
- Git、symlink、ファイル削除を伴う検証は、実データではなく隔離した環境で行う。
- 検証できなかった項目は、完了報告に理由とともに記載する。

## コマンド実行とsandbox

- ユーザーはGit Bashを利用しているため、特別な理由がない限りbash前提の単純なコマンドを使う。
- 読み取りや探索は `rg`、`sed`、`git diff` などの単発コマンドを優先し、パイプ、リダイレクト、`&&`、`;`、`$(...)` などsandbox判定が複雑になる書き方を避ける。
- 書き込み先は原則としてワークスペース配下に限定し、ホームディレクトリ、グローバルキャッシュ、外部tempなどへ書き込む操作は事前に理由を確認する。
- `.git` にロックファイルやrefsを書き込む操作（例: `git add`、`commit`、`checkout`、`pull`、`merge`）や、ネットワークアクセスを伴う操作は、失敗してから報告するのではなく、実行前に許可が必要か判断する。
- `make` 経由の検証でもGoや外部ツールがワークスペース外のcacheへ書き込む可能性があるため、失敗時は原因を確認し、必要ならcacheをワークスペース内へ寄せるか許可を求める。

## スキル

再利用可能な作業手順は `.claude/skills/` に定義されている。
タスクがスキルの対象に合致する場合はスキルの手順に従うこと。

## 計画・メモの使い分け

- そのセッション内で完結する短い計画や調査は、AIエージェント内部で管理し、ファイルを作成しない。
- 複数セッションにまたがる作業、別エージェントへの引き継ぎ、ユーザーが成果物として求めた計画は `.agents/plan/` に保存する。
- 複数タスクをまとめるロードマップ的な plan は `_` 接頭辞を付ける（例: `_priority.md`）。ソート順で先頭に来るため、個別タスクの plan と視覚的に区別できる。
- 後続作業で再利用する調査結果や判断記録は `.agents/note/` に保存する。
- 対応が完了した plan や役割を終えた note は `.agents/archive/` に移動する。`/archive` スキルで判定・移動できる。
- 一時的な進捗、ツール出力、会話だけで十分な情報は `.agents/` に残さない。

## ビルド・テストの実行

`go test` や `go build` を直接実行しない。必ず Makefile 経由で実行すること。

```bash
make fmt    # gofumpt
make lint   # golangci-lint
make test   # go test（GOTMPDIR=dist/ を設定済み）
make build  # go build → dist/dotfile
```

Makefile は `GOTMPDIR` を `dist/` に設定しており、ビルド・テストの一時ファイルがワークスペースを汚さないようになっている。`go test ./...` を直接叩くと一時ファイルがワークスペース内に散らばる。
